### 总体思路

* 将事物保持在"培育器"状态（以防它们死亡），直到它们成为官方项目。这将防止主要项目区域变得混乱，并允许清理旧的、不再使用的项目。
* 为所有这些构建移动支持？

[ostag](./index.mdx#about)

* 需要对所有这些进行监控  
  * （Nagios 和 MRTG... 谁来处理通知页面？）  
  * ZABBIX... ZABBIX... ZABBIX.
* 注册页面来开始这一切

* 单点登录

  * 我觉得还是使用基本的 LDAP。OpenID 很酷，crowd 是针对 Atlassian 的。OpenSSO 不错，但它们都是为超过 50 万用户设计的，并且具有相应规模的维护工作。
* 良好的备份解决方案  
  * [bacula](http://www.bacula.org/)  
  * [rsnapshot](http://www.rsnapshot.org/) 简单易用
* 良好的速度/性能  
  * 很多是 Java 的...我们需要大型服务器、大量内存和高速硬盘驱动器来应对这些...

你可以在一个Glassfish实例中运行所有这些，并且可以通过单个JVM运行所有这些...如果负载过重，只需添加一个带有Glassfish的第二个服务器，并将其集群化。它本身支持这个功能。

### 我们想要支持的功能...

SSO (使用 Crowd 的基于 ldap 的 OpenID) 来支持以下服务 \[1\]

* 子域名的私人邮件  
   * 普通的邮件（如咨询）  
   * 私人的邮件（只是重定向到个人邮件）
* 自动为项目创建子域名
* WIKI  
   * [MediaWiki](http://www.mediawiki.com/)  
   * [Confluence](http://www.confluence.com/)  
   * [XWIKI](http://www.xwiki.org/) 提供良好的多语言支持 .. ldap 授权和大量的模块。  
   * 可能还有一个预先构建的门户/模板，包括诸如“我的项目中有什么更新”之类的内容。
* Subversion
* OpenGROK
* Fisheye
* Crucible
* Jive Forums?
* 邮件列表  
   * [OpenIDEnabled](http://openidenabled.com/unsupported-software/)  
   * [Nabble2](http://www.nabble2.com/)

其中很多可以通过一个新项目的 Apache 模板来实现，该模板也可以放入 OpenID 框架中.. (Apache 支持 OpenID)

* 基于名称自动生成 IRC 频道  
   * 与 ChanServ 注册  
   * 为用户生成添加 IRC Web 客户端的链接/说明等
   * 等等。
* 实时电话通信的自动设置  
   * 预定义/自动生成会议室  
   * 可能会给人们提供自己的上下文，映射到他们的域名
